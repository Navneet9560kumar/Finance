# Finance Dashboard Backend API

A robust, secure, and logically structured backend system for a Finance Dashboard. This API manages users, enforces Role-Based Access Control (RBAC), handles financial records (income/expenses), and provides powerful aggregation-based summaries for dashboard analytics.

## тЬи Core Features

- **User & Role Management** тАФ Secure registration and login with encrypted passwords. Three roles: Viewer, Analyst, and Admin. Admins can manage all users including activating or deactivating accounts.
- **Role-Based Access Control (RBAC)** тАФ Middleware-level protection ensuring users only access what their role permits. Inactive users are blocked at the middleware level.
- **Financial Records Management** тАФ Full CRUD operations for income and expense records.
- **Advanced Search & Filtering** тАФ Powerful query support including Text Search (partial match), Date Ranges, Amount Ranges, and Dynamic Sorting (Highest amount, Newest, etc.).
- **Dashboard Analytics** тАФ MongoDB Aggregation pipeline to calculate total income, total expenses, net balance, category-wise totals, recent activity, and monthly trends in a single optimized API call.
- **Data Validation & Error Handling** тАФ Input validation on all endpoints, standard HTTP status codes, and a global error handling middleware to prevent server crashes.

## ЁЯТб Design Decisions

- **Cloud Persistence** тАФ MongoDB Atlas used for production-ready cloud storage.
- **Aggregation for Performance** тАФ MongoDB `$aggregate` pipeline used for all dashboard calculations at the DB level, avoiding unnecessary data transfer.
- **Advanced Filtering** тАФ Integrated regex matching and range queries directly into the Mongoose `find` operators.
- **Global Error Handling** тАФ Centralized `errorMiddleware` catches all errors and returns clean JSON responses with appropriate status codes.
- **Inactive User Blocking** тАФ The `protect` middleware fetches the user from DB on every request to check `isActive`, so deactivated users are blocked immediately even if their token is still valid.
- **asyncHandler Wrapper** тАФ All async route handlers are wrapped to forward errors to the global error handler without try-catch in every route.
- **Input Validation** тАФ Email format (regex) and password length (min 6 chars) validated on registration.

## тЪая╕П Assumptions Made

- Any user can self-register with the default **Viewer** role. Only an Admin can upgrade roles.
- Soft delete is not implemented; `DELETE` permanently removes the record.
- No pagination is implemented on record listing (all records matching filters are returned).
